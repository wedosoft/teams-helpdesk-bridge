markdown# Teams 연계 컨설팅 분석 - ChatGPT 의견

## 핵심 판단

대표님이 정리해주신 판단 방향이 거의 정답에 가깝습니다. 이 고객(포스코인터내셔널 법무실)이 원하는 것을 다음과 같이 해석합니다:

- **엔드유저(현업)**: Teams/Outlook에서 벗어나고 싶지 않음. 이미 요청이 그 채널로 흘러들어오는 게 현실
- **법무실**: '케이스(검토건)' 단위로 접수/배정/진행/완료/이력/통계를 한 곳에서 보고 싶음
- **특히**: 상위자가 진행현황/완료현황/총 건수, 실원별 처리량을 보고 싶고, 최근엔 가중치(1~5) 같은 난이도/중요도 지표까지 요구

## 추천 구조

"**Teams=프론트, Freshdesk Omni=백오피스(법무실 운영)**"

메일에는 Freshdesk가 직시되지 않지만, EXO Help가 "Teams ↔ Freshdesk Omni 채널을 브릿지로 실시간 중계"한 사례이므로, 가장 자연스러운 구현입니다.

## 1) 아키텍처 판단

### SSOT(Single Source of Truth)는 Freshdesk Omni

- 업무 접수/커뮤니케이션 창구: Teams(및 Outlook)
- 법무실이 Teams를 선호한다면 Teams에 '탭(Tab) 대시보드'를 얹어도 됨 (그래도 데이터는 Freshdesk가 원장)

이렇게 잡으면, 고객이 말하는 "Teams 기반으로 바꾼다"와 대표님이 말하는 "법무실은 Freshdesk에서 관리한다"를 충돌 없이 동시에 만족시킬 수 있습니다.

## 2) Teams에서 "화면 이동 없이" 가능한 한계

**결론**: Teams '채팅창만'으로 모든 업무처리는 불가능합니다.
- 첨부/검색/대시보드/권한별 뷰 같은 건 채팅 UI가 구조적으로 약합니다.

하지만 **Teams를 벗어나지 않고(=Teams 안에서) 대부분을 처리**하도록 만드는 건 가능합니다.

### Teams에서 강한 것(적합)

- 요청 접수 폼: Adaptive Card로 제목/내용/가중치/열람자/첨부 링크 입력
- 상태 조회/알림: "접수됨/배정됨/추가정보 요청/완료" 같은 이벤트 알림
- 짧은 Q&A: 법무가 추가 질문 → 요청자가 답변
- 승인/확인류: 상위자에게 "중요도 5 / 기한 D+3" 같은 확인 버튼

### Teams에서 약한 것(여기서 '탭'이 필요)

- 여러 건을 리스트로 탐색/필터/검색 (내 요청 목록, 기간별 통계)
- 실원별/부서별 처리량 + 가중치 합산 대시보드
- 첨부파일 관리(버전/권한/다운로드)
- 케이스 상세 이력(댓글/내부노트/외부노트/메일 스레드) 심층 조회

### 추천 UX: "3면(面) 구성"

1. **엔드유저용 Bot 채팅**: "검토요청 올리기 / 진행상황 보기 / 추가정보 제출"
2. **엔드유저용 Personal Tab(내 요청함)**: 목록/필터/상태/이력(Teams 안에서 웹 탭)
3. **법무실 운영**: Freshdesk Omni(원장), 필요하면 법무용 Teams 탭으로 지표만 노출

이렇게 하면 대표님이 원하시는 "가급적 화면 이동 없이"를 현실적으로 달성합니다. (Teams 내부에서만 움직이게)

## 3) 고객 요구사항을 "케이스 관리"로 재정의

메일에 나온 요구를 기능으로만 나열하면 산으로 갑니다. 핵심을 케이스(검토건) 라이프사이클로 재정의해야 합니다.

### 제안하는 케이스 상태(예시)

- 접수(NEW) → 분류/가중치(1~5) → 담당 배정(ASSIGNED) → 검토중(IN PROGRESS) → 추가정보 요청(WAITING ON REQUESTER) → 검토완료(RESOLVED/CLOSED)

### 역할: 최소 4개로 쪼개기

- 요청자(현업)
- 열람자(CC)
- 담당 변호사/실무자
- 상위자(리더)

이 프레임으로 잡으면, "열람자 추가", "가중치", "요청자가 진행 확인" 같은 요구가 자연스럽게 케이스 필드/권한/알림 정책으로 내려옵니다.

"메일/채팅으로 주고받은 내용을 담당직원이 올릴 수 있나?"는 케이스에 증빙/커뮤니케이션 로그를 적재하는 문제로 정리됩니다.

## 4) 추천 아키텍처: Exo Help 패턴을 "Legal Case"로 확장

### 권장 구조(Production 기준)

**Teams App**
- Bot: 요청 접수/알림/간단 질의응답
- Personal Tab: 내 요청 목록/상태 (옵션)
- Message Extension: "이 메시지를 검토건으로 등록"

**Bridge Server**
- Teams ↔ Freshdesk Omni API/Webhook 중계
- 케이스 매핑 저장(Teams conversation/message id ↔ Freshdesk ticket/case id)
- 권한/감사로그/재전송(실패시) 처리

**Freshdesk Omni**
- 케이스(티켓) 원장
- 사용자/조직/그룹/담당자 배정
- 커뮤니케이션 이력(메일/노트/답변)
- 리포팅(가중치 커스텀 필드 기반)

### "Outlook/Teams 기반으로 들어오는 건" 처리 방식

**A안(가장 빠르고 안전)**: "Teams에서 '등록'을 표준화"
- 현업/법무 모두, 들어온 메일/채팅을 보고 **Teams Bot에 '등록'**해서 케이스로 만든다.
- 장점: 구현/운영 단순, 실패율 낮음
- 단점: '등록'이라는 한 번의 액션은 사람 손이 탐

**B안(자동화 지향)**: Outlook(공용메일함) → 케이스 자동 생성 + Teams 알림
- 장점: 누락 최소화
- 단점: 고객 테넌트 권한/보안/예외처리 때문에 POC 난이도 상승

**권장**: POC는 A안으로 확실히 성공시키고, 본사업에서 B안을 옵션으로 확장 제안

## 5) 2~3일 "초간단 POC"로 12/22 미팅에서 보여줄 데모 시나리오

POC 범위는 욕심내면 망합니다. 임팩트 있는 3개 장면만 보여주시면 됩니다.

### POC에서 꼭 보여줄 3개 장면(추천)

**장면 1: Teams에서 검토요청 생성(가중치 포함)**
- Teams Adaptive Card: 제목/내용/가중치(1~5)/열람자(텍스트라도)/첨부(링크)
- 제출하면: "케이스 번호 / 현재 상태 / 담당(미배정)" 카드로 회신

**장면 2: Freshdesk에서 법무가 상태 변경/댓글 → Teams로 실시간 알림**
- 예: "추가정보 요청"으로 바꾸면 Teams에 알림 카드가 오고 요청자는 Teams에서 "추가정보 제출" 버튼으로 답변
- → 케이스에 노트/댓글로 적재

**장면 3: 상위자 관점 '건수 + 가중치 합' 간단 대시보드**
- Teams 탭(또는 웹페이지)로 "실원별 진행중/완료/가중치합"만 보여주면 됩니다.
- 이게 메일에 나온 "단순 건수 말고 가중치로 보고 싶다" 요구를 정면으로 해결합니다.

### POC 구현 포인트(최소 구현)

- Freshdesk 티켓에 커스텀 필드로 가중치(1~5) 저장
- Bridge 서버에 매핑 테이블(Teams conversation ↔ ticket id)만 저장
- Freshdesk Webhook(또는 이벤트) → Bridge → Teams proactive message

POC는 "완벽한 권한/감사로그/첨부 정책"까지 할 필요 없습니다. 대신 "이 구조로 된다"를 보여주는 게 목적입니다.

## 6) 12/22 미팅에서 컨설팅을 '성공'으로 끌고 가는 접근

미팅에서 기술설명부터 하면 고객은 바로 지칩니다. 다음 순서로 끌고 가시는 게 좋습니다:

**업무/운영 질문 → 구조 제안 → POC 데모 → 다음 단계 합의**

### (1) 고객 Pain을 한 문장으로 정리해서 시작

"요청 채널이 Outlook/Teams로 분산됐고, 법무는 진행현황/이력/통계를 한 곳에서 못 봐서 관리가 안 된다."

"그래서 'Teams로 접수는 통일, 케이스 관리는 하나의 원장'이 필요하다."

(메일 내용 그대로라 고객 공감이 바로 나옵니다.)

### (2) 선택지를 2개만 제시(많이 주면 결정 못 합니다)

**선택 1**: Teams=프론트, Freshdesk=원장(추천)

**선택 2**: Teams/PowerPlatform 중심으로 별도 원장 구축(비용/기간↑)

거의 대부분은 선택 1로 갑니다. (이미 EXOHelp라는 내부 성공사례가 있으니까요.)

### (3) POC를 보여주고 "다음 단계"를 아주 구체적으로 제안

"POC는 접수/상태알림/가중치 대시보드까지 보여드렸고, 다음 단계는 (a) 역할/권한 확정, (b) Outlook 유입 처리 방식 결정, (c) 데이터 보관/보안 정책 합의입니다."

## 7) 제가 대표님께 권하는 결론(한 줄)

'법무앱을 새로 만들자'가 아니라, **'Teams를 디지털 프론트도어로 만들고 Freshdesk Omni를 법무 케이스 원장으로 삼는' 컨설팅**으로 들어가시는 게 가장 빠르고 안전합니다.

원하시면 위 POC 데모를 "어떤 화면/어떤 멘트/어떤 순서로 보여주면 고객이 바로 이해하는지"까지, **12/22 발표용 데모 스크립트(3분/7분 버전)**으로도 정리해드리겠습니다.

markdown# Teams 연계 컨설팅 분석 - Gemini 의견

## 전략 개요

업로드해주신 메일 내역과 위두소프트의 기존 솔루션(EXO Help, Freshchat Bridge)을 바탕으로, 12월 22일 월요일 미팅을 위한 컨설팅 전략 및 POC(개념 증명) 방향성을 정리합니다.

**핵심**: "End-User는 Teams에서 머물고, 법무팀은 전문 솔루션(Freshdesk Omni)에서 관리하는 구조"가 이 프로젝트의 핵심 성공 요인입니다.

## 1. 고객(포스코인터내셔널 법무실) 요구사항 분석

메일 내역을 종합해보면, 고객의 Pain Point와 니즈는 다음과 같이 명확합니다.

### 현재 문제

법무앱, 아웃룩, 팀즈로 소통 채널이 파편화되어 있어 관리가 어렵고, 특히 아웃룩/팀즈 요청이 증가하여 추적 관리가 안 됨.

### 핵심 목표

**1. 채널 통합**
- 기존 법무앱 폐쇄 후 팀즈/아웃룩 기반으로 프로세스 일원화

**2. 데이터 관리**
- 채팅이나 메일로 들어온 건도 '시스템'에 등록되어 진행 현황, 완료 현황, 통계가 나와야 함

**3. 가중치(Weight) 관리**
- 단순 건수 외에 업무 난이도(1~5점)를 부여하여 상위자가 실질적인 업무량을 파악하고자 함

**4. 편의성**
- 현업 부서는 팀즈에서 쉽게 질의하고, 법무 담당자는 팀즈/메일 내용을 쉽게 시스템에 이관(등록)하길 원함

---

## 2. 아키텍처 판단 및 MS Teams 통합 가능성

### 구상하신 구조
[Teams (User) ↔ Middleware (Bridge) ↔ Freshdesk Omni (Agent)]

이사님께서 구상하신 이 구조는 고객의 니즈에 정확히 부합합니다.

### Q. MS Teams에서 업무 시스템 연계가 어디까지 가능한가? (화면 이동 최소화)

전문 컨설턴트 입장에서 볼 때, Teams의 **'앱 기능(App Capabilities)'**을 적극 활용하면 화면 이탈 없는 프로세스 구현이 가능합니다.

#### Message Extension (메시지 확장)

**기능**: 채팅창 옆의 `...` 버튼이나 입력창 하단 버튼 활용

**적용**: 법무 담당자가 현업에게 받은 팀즈 1:1 채팅 메시지를 클릭하여 "법무 검토 요청 등록" 버튼을 누르면, 해당 대화 내용이 자동으로 Freshdesk 티켓으로 생성됩니다. (고객의 요구사항인 "담당자가 선별하여 등재" 완벽 해결)

#### Adaptive Cards (적응형 카드)

**기능**: 봇이 사용자에게 폼(Form)이나 버튼이 포함된 카드를 전송

**적용**: 사용자가 봇에게 "법무 검토 요청"이라고 말을 걸면, 봇이 입력 양식(제목, 내용, 기한 등)을 카드로 띄워줍니다. 사용자는 팀즈 채팅방 내에서 폼을 작성하고 '제출'을 누릅니다.

#### Personal App (개인 앱 탭)

**기능**: 팀즈 좌측 메뉴에 'Legal Help' 아이콘 고정

**적용**: 클릭 시 **"나의 요청 현황(Dashboard)"**을 보여줍니다. 사용자는 별도 웹사이트 접속 없이 여기서 진행 상태를 확인합니다.

## 3. 컨설팅 전략 및 접근 제안

고객에게 제안할 핵심 가치는 **"Invisible System(보이지 않는 시스템)"**입니다. 현업과 법무 담당자는 익숙한 팀즈/아웃룩을 쓰지만, 뒤단에는 강력한 티켓팅 시스템이 돌아가는 구조입니다.

### 전략 1: "채팅이 곧 업무가 된다" (Seamless Workflow)

**현업**: "팀즈에서 말을 걸거나 봇 메뉴를 누르면 접수가 끝납니다."

**법무**: "메일이나 채팅으로 온 건을 '클릭 한 번'으로 시스템에 등록합니다." (Outlook Add-in 및 Teams Message Extension 제안)

### 전략 2: "데이터 기반의 법무 성과 관리" (Weighting System)

조영탁 리더님이 강조한 **가중치(1~5)** 기능은 Freshdesk의 **'Custom Fields(사용자 정의 필드)'**로 즉시 구현 가능함을 강조합니다.

단순 건수가 아닌 '가중치 합산 점수'로 팀원들의 로드 밸런싱을 시각화할 수 있음을 어필하십시오.

### 전략 3: "협업 기능 강화"

**열람자 추가**: 문의 주고받는 건에 열람자 추가 → Freshdesk의 'CC' 기능 또는 'Collaborator' 기능으로 구현

**검토 의견 확인**: 티켓 내의 'Public Note'가 팀즈 봇을 통해 사용자에게 알림으로 전송됨

## 4. 대기업 법무팀 일반 프로세스 및 매핑

## 추가 상세 분석

### 티켓 관리 VS 채팅 관리

**핵심**: "사용자는 채팅(비동기 대화)처럼 느끼지만, 시스템은 이를 정형화된 티켓(문서)으로 처리한다"

이를 위해 중간에 위치한 Freshchat Bridge(미들웨어)가 통역사 역할을 수행해야 합니다.

#### 구조적 개념

**MS Teams (User View): "대화형 인터페이스"**
- 사용자는 복잡한 게시판 양식을 보지 않습니다. Legal Bot과의 1:1 대화방이 곧 접수 창구입니다.
- 하지만 단순히 텍스트만 치면 정보가 누락되므로, **Adaptive Card(적응형 카드)**를 통해 필수 입력값(제목, 유형, 기한)을 폼(Form) 형태로 받습니다.

**Freshdesk Omni (Agent View): "티켓형 인터페이스"**
- 법무 담당자는 채팅창을 띄워놓고 대기하지 않습니다. 접수된 건은 '티켓(Ticket)' 형태로 목록화됩니다.
- 사용자가 입력한 Adaptive Card의 데이터는 Freshdesk 티켓의 **'사용자 정의 필드(Custom Fields)'**에 자동으로 매핑되어 저장됩니다.

### 데이터 흐름 (Data Flow)

#### Scene 1: 현업의 간편한 요청 (Inbound Flow)

1. **Trigger**: 사용자가 Teams에서 "검토 요청" 입력
2. **Bridge Action**: 미들웨어가 키워드를 인식하고 Teams Adaptive Card JSON을 반환
3. **User Input**: 사용자가 카드 내 제목, 내용, 파일, 희망기한 입력 후 [제출] 클릭
4. **Creation**:
   - 미들웨어가 Freshdesk API (POST /api/v2/tickets)를 호출
   - Teams의 User ID(Email)를 Freshdesk의 **Requester(요청자)**로 매핑
   - 매핑 키: MS Teams Email = Freshdesk Contact Email
5. **Feedback**: 티켓 생성 성공 시, 리턴받은 Ticket ID (#101)를 포함하여 Teams 봇이 메시지 전송
   - "접수가 완료되었습니다. (접수번호: #101)"

#### Scene 2: 법무팀 처리 및 알림 전송 (Outbound Flow)

1. **Agent Action**:
   - 법무 담당자가 Freshdesk에서 티켓 #101을 열고 **[가중치: 5]**로 설정
   - '공개 답장(Public Reply)' 기능으로 "계약서 초안 파일이 누락되었습니다. 전달 부탁드립니다."라고 입력

2. **Trigger (Automation)**:
   - Freshdesk의 'Automation rules for ticket updates' 기능 활용
   - 조건: "답장이 추가되었을 때(Note Added)" 또는 "상태가 변경되었을 때"
   - 액션: Webhook 호출 → Freshchat Bridge 서버로 데이터 전송
   - Payload: `{ "ticket_id": 101, "user_email": "user@poscointl.com", "comment": "계약서 초안..." }`

3. **Bridge Action**:
   - 미들웨어가 Payload를 수신
   - user_email을 기반으로 Teams Proactive Message (봇이 먼저 말을 거는 API) 실행

4. **User View**:
   - Teams 봇 채팅방에 새 메시지 도착
   - "[법무팀 알림] 담당자가 의견을 남겼습니다: 계약서 초안 파일이 누락되었습니다..."

#### Scene 3: 이메일의 티켓 변환 및 매핑 (Email Integration)

**Forwarding Logic (전달)**
- 법무 담당자가 받은 메일을 legal@support.wedosoft.net (Freshdesk 수신용 주소)로 전달(Fwd)
- Freshdesk의 Forwarding Rule을 사용하여, 메일 본문 내의 "From: user@poscointl.com" 텍스트를 파싱해 원래 보낸 사람을 티켓 요청자(Requester)로 자동 지정

**Outlook Add-in (권장)**
- Outlook 우측 패널에서 Freshdesk Add-in 실행
- 해당 메일을 보면서 바로 "Convert to Ticket" 버튼 클릭
- 이때 속성창에서 바로 [가중치: 3] 설정 가능

**결과**
- 이메일로 접수된 건도 Freshdesk에 티켓으로 생성됨
- 이후 처리는 Scene 2와 동일하게 진행 (답변 시 메일 회신 or 팀즈 알림 설정 가능)

### 대화 매핑 (Conversation Mapping) 전략

"어떤 채팅이 어떤 티켓에 속하는가?"를 관리하는 것이 가장 중요합니다.

#### Stateless 방식 (간편 구현, POC용)

- 사용자가 봇에게 말을 걸면 무조건 **"새로운 요청입니까, 기존 건에 대한 문의입니까?"**를 묻는 카드를 띄웁니다
- 기존 건 선택 시: 진행 중인 티켓 리스트를 보여주고 선택하게 함. 선택 후 입력한 대화는 해당 티켓의 '비공개 메모(Private Note)' 또는 **'답장'**으로 API 전송

#### Stateful 방식 (고도화)

- Bridge 서버 DB에 `{ User_ID : Active_Ticket_ID }` 세션을 저장
- 가장 최근에 생성된(혹은 대화 중인) 티켓이 'Open' 상태라면, 사용자가 Teams에 치는 모든 채팅을 해당 티켓의 대화 로그로 자동 적재
- 티켓이 'Closed' 되면 세션 종료

#### 12/22 미팅용 POC 제안

**Stateless 방식(매번 선택 혹은 신규 생성)**이나, 가장 단순하게 **"채팅 입력 = 신규 티켓 생성"**으로 보여주고, 답변이 오는 것만 시연하는 것이 리스크가 적고 임팩트는 확실합니다.

## 실시간 채팅 vs 알림/접수용

### 왜 "실시간 채팅"이 아닌 "알림/접수용"인가?

법무 검토는 콜센터처럼 즉답할 수 있는 업무가 아닙니다.

- **사용자 입장**: 익숙한 채팅창(Teams)에서 편하게 말을 걸어 접수하니 '채팅'처럼 느껴집니다
- **법무팀 입장**: 실시간으로 대기할 필요 없이, 접수된 건을 보고 여유 있게 검토 후 답변하면 됩니다
- **결과**: 겉으로는 채팅이지만, 실제로는 **'비동기 티켓 시스템'**으로 동작하여 양쪽의 니즈를 모두 충족합니다

### 왜 앱을 2개 만들 필요가 없는가?

#### 전략 A: "Teams 앱 하나로 Outlook까지 커버" (최신 방식)

최근 MS는 'Teams 앱'을 만들면 Outlook과 Microsoft 365 앱에서도 그대로 작동하도록 지원합니다.

- 우리가 개발할 Teams 봇/탭 앱(Manifest) 하나만 배포하면, Outlook 좌측 메뉴에도 동일한 아이콘이 생기고 똑같이 작동합니다
- 즉, **"One Code, Run Anywhere"** 전략이 가능합니다

#### 전략 B: "Outlook은 개발 없이 기능만 활용" (POC 추천 전략)

당장 12월 22일 미팅을 위한 POC(개념 증명) 단계에서는 굳이 Outlook 앱을 개발하거나 연동할 필요조차 없습니다.

- **Teams**: 봇 개발에 집중 (여기가 메인)
- **Outlook**: '이메일 자동 전달(Forwarding)' 기능만 설정
- 법무팀이 받은 메일을 support@... 주소로 전달만 하면 자동으로 티켓이 생성되므로, 별도의 아웃룩 앱 개발 없이도 **"아웃룩 연동도 됩니다"**라고 시연할 수 있습니다

**개발(Dev)**: 오직 MS Teams 봇 하나만 만드십시오. (Adaptive Card로 접수 받고, 알림 주는 기능)

**설정(Config)**: Outlook은 개발하지 말고, Freshdesk의 **'이메일→티켓 변환 규칙'**만 세팅하십시
